#+title: Some tools

This repo collects some opinionated tools I've written to make my life just a little easier.  The tools fall into these categories:

- Shell (usually bash) scripts (under [[file:scripts/]])
- Python modules with CLI entry points (under [[file:src/tools/]])

This package is written for my use, but others are free to use it according to its [[file:LICENSE][GPLv3+]] licensing.  I am receptive to bug fixes and extending functionality with good ideas but reserve the right to ignore input on a case-by-case basis.  If you use this package, see also the section [[Cautions]].

I also have a [[https://github.com/brettviren/per][per]] package that aggregates multiple individual packages via =uv=, including many I developed but which are too "big" to simply be a part of =tools=.

* Install

#+begin_center
*Beware, this package has nothing to do with the [[https://pypi.org/project/tools/]["tools" package on PyPI]].*
#+end_center

As an end-user:
#+begin_example
uv tool install git+https://github.com/brettviren/tools
uv tool upgrade tools
#+end_example

As a developer:
#+begin_example
git clone git@github.com:brettviren/tools.git
cd tools
uv tools install -e .
uv tools install -e --upgrade --force
tools [...]
#+end_example
See also =install-hooks= under [[Cautions]].

Optionally, install man pages and completions:

#+begin_example
$ tools manpages     # May need to add to your MANPATH
$ tools completions  # Tested on fish, your shell mileage may vary
#+end_example

* Multiple accounts

Some of the tools (clones, manycron) only make sense to use when you have
multiple user accounts across many computers.  To mange the configuration of
these tools and various account configurations, using [[https://github.com/richih/vcsh][vcsh]] has served me well
for many years.  In particular =clones= can handle vcsh (and [[https://www.passwordstore.org/][pass]]).  For multiple
different git clones in one account, take a look at [[https://myrepos.branchable.com/][mr / MyRepos]].

* Cautions

I am not responsible if this package burns down all that you hold dear.  That said, I use these tools daily.  However, my opinion on the "right" way to do things may not match yours.  This section collects potential areas where our differing opinions may be important for you to consider.

** bwrap-cwd policy in source

The [[file:.bwrap-cwd]] in this modify the policy enacted by =bwrap-cwd= when that command is run inside this directory.  These files are location specific and this file is not honored when using =bwrap-cwd= from other.  This file exists only in the source and is not installed.  Run =bwrap-cwd= for more info on how its policy is constructed.

** Auto-reinstall on pull

When developing across several clones (see the =clones= tool), each clone needs
=uv tool install -e . --upgrade --force= re-run after a pull to expose new or
changed tools on =$PATH=.  This repo ships git hooks under =.githooks/= that do
this automatically after a pull touches =pyproject.toml=, =uv.lock= or =src/=.

The hooks are version-controlled and you must take action to enable them to be
used.  Enabling them sets =core.hooksPath=, which is per-clone local config and
cannot itself be tracked.  Enable them with:

#+begin_example
$ scripts/install-hooks here   # this clone
$ scripts/install-hooks all    # every clone in this package's clones group
#+end_example

=install-hooks all= must be run from within a clone of this package: it fans
out via =clones run git config core.hooksPath .githooks=, letting =clones=
infer the group from the current directory so it only touches sibling clones of
this package — not every configured group (many of which are not Python
packages).  Running that =clones run= line directly is equivalent.

Note: the hooks also fire on the remote side during =clones sync= (which pulls
over ssh).  If =uv= is not on the non-interactive =$PATH= there, the hook prints
a notice and skips; adjust the =PATH= fallback in =.githooks/_install= if your
=uv= lives outside =~/.local/bin=.

The =install-hooks= will also configure smudge filters needed to work-around GitHub's imperfect org-mode renderer.
